# Руководство по контрибьютору

Спасибо за интерес к этому проекту! Это обучающий репозиторий для изучения Spring Boot.

## Как помочь развивать проект?

### Типы контрибьютов

1. **Добавление новых примеров и проектов**
   - Новые тематические приложения (REST API, БД, Security и т.д.)
   - Демонстрационные проекты на определенные темы

2. **Улучшение документации**
   - Исправление опечаток
   - Добавление примеров кода
   - Уточнение инструкций

3. **Исправление ошибок**
   - На ошибки в примерах кода
   - На неработающие примеры
   - На устаревшую информацию

4. **Добавление примеров кода**
   - В папку `docs/examples/`
   - С пояснениями и комментариями

## Процесс контрибьютора

### 1. Fork репозитория
```bash
# На GitHub нажми кнопку "Fork"
# Потом клонируй свой форк:
git clone https://github.com/YOUR_USERNAME/spring-boot-education.git
cd spring-boot-education
```

### 2. Создай ветку для своих изменений
```bash
git checkout -b feature/add-database-example
# или
git checkout -b fix/typo-in-docs
```

### 3. Делай изменения

#### Если добавляешь новый проект:
- Следуй структуре из existing проектов
- Добавь `build.gradle.kts`
- Создай `application.properties`
- Напиши примеры кода с комментариями
- Обнови `docs/PROJECT_STRUCTURE.md`

#### Если улучшаешь документацию:
- Используй Markdown
- Добавляй примеры кода для каждого объяснения
- Проверяй работоспособность примеров

#### Стиль кода:

```java
// Java классы - PascalCase
public class UserService {
    
    // Методы - camelCase
    public User getUserById(Long id) {
        // 4 пробела для отступа
        if (id == null) {
            throw new IllegalArgumentException("ID cannot be null");
        }
        return repository.findById(id).orElse(null);
    }
}
```

- Отступ: 4 пробела (настроено в `.editorconfig`)
- Максимальная длина строки: 120 символов
- Используй `var` только когда тип очевиден
- Добавляй JavaDoc для public методов

### 4. Тестирование

```bash
# Проверь, что все собирается
./gradlew build

# Запусти тесты
./gradlew test

# Если добавляешь новый проект, запусти его
./gradlew :new-project:bootRun
```

### 5. Коммиты

```bash
# Используй понятные сообщения коммитов
git add .
git commit -m "Add REST API example with error handling"
git commit -m "Fix: correct typo in installation guide"
git commit -m "Docs: add Spring Data JPA example"
```

**Префиксы коммитов:**
- `Add:` - новая функция или проект
- `Fix:` - исправление ошибки
- `Docs:` - улучшение документации
- `Refactor:` - переструктурирование кода
- `Update:` - обновление зависимостей или конфигов

### 6. Push и Pull Request

```bash
git push origin feature/add-database-example
```

В GitHub:
1. Открой Pull Request
2. Напиши описание: что добавил, почему, как тестировать
3. Дождись review

## Требования к Pull Request

- ✅ Код следует стилю из `.editorconfig`
- ✅ Все тесты проходят
- ✅ Документация обновлена
- ✅ Нет merge конфликтов
- ✅ Понятное описание в PR

## Структура PR описания

```markdown
## Описание
Что делает этот PR?

## Прикреплено к Issue
Если фиксит issue: Fixes #123

## Как тестировать?
1. Запусти ./gradlew build
2. Запусти ./gradlew :project-name:bootRun
3. Откройся http://localhost:8080/api/...

## Скриншоты (если applicant)
[Скриншоты результата]

## Чек-лист
- [x] На коде нет ошибок
- [x] Код соответствует стилю
- [ ] Добавлены тесты
- [ ] Обновлена документация
```

## Что НЕ надо делать

- ❌ Не добавляй файлы IDE конфигов (`.idea`, `.vscode` локальные)
- ❌ Не коммитай сгенерированные файлы (`build/`, `target/`)
- ❌ Не меняй общую конфигурацию без обсуждения
- ❌ Не используй в примерах пароли или ключи

## Вопросы?

- Открой Issue для обсуждения перед большими изменениями
- Комментируй в PR если непонятно
- Посмотри существующие примеры для вдохновения

Спасибо за контрибьют! 🎉
